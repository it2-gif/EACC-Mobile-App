EACC Chat Mobile App - Project Progress

Last Updated: 2026-06-18

Project Goal
============

Build a professional Flutter chat application for EACC with three roles:

1. Student
2. Teacher
3. Admin

Privacy requirements:

- A student sees only their courses and their own chat thread.
- A student never sees another student's conversation.
- A teacher sees only their courses and the student threads in those courses.
- An admin can monitor all authorized courses, users, and chats.
- Production permissions must be enforced by Firebase and the backend, not only
  by the Flutter interface.

Next Phase: Production Data Model
=================================

The next task is to define the permanent data model before authentication and
LMS integration.

The model must define:

1. Users
   - id
   - lms_user_id
   - role
   - name
   - email
   - status

2. Courses
   - id
   - lms_course_id
   - name
   - category
   - status

3. Course memberships
   - user_id
   - course_id
   - role
   - active

4. Chat threads
   - id
   - course_id
   - student_id
   - last message metadata
   - timestamps

5. Messages
   - id
   - thread_id
   - sender_id
   - sender_role
   - type
   - text
   - attachment metadata
   - timestamps

6. Device tokens
   - user_id
   - FCM token
   - platform
   - last active time

7. Audit records
   - admin actions
   - security events
   - relevant timestamps

Important design rules:

- Use stable LMS IDs and database IDs, never names as identity.
- A thread belongs to exactly one student and one course.
- Membership data determines access to a course.
- The backend is the trusted source for roles and permissions.
- Flutter must not choose or grant a role in production.

Security Notes
==============

- Use Firebase custom claims and server-controlled memberships.
- Validate attachment type and size on both client and server.

Secure Student and Teacher Authentication
=========================================

- Student and teacher LMS login is connected to the NestJS backend.
- Authenticated LMS users and active courses are synchronized to PostgreSQL.
- The backend creates Firebase custom tokens using stable UIDs:
  - student:<lms_user_id>
  - teacher:<lms_user_id>
- Signed claims contain the role, LMS user ID, display name, app user ID, and
  compact active-course ID list.
- Flutter signs into Firebase only after LMS authentication succeeds.
- LMS passwords are never stored by Flutter.
- Non-secret session/course data is restored only while the matching Firebase
  identity remains signed in.
- Persisted app sessions are rejected after 12 hours when the app starts, and
  logout clears both Firebase and saved session data.
- Firestore rules restrict students to their own course thread.
- Teachers can access threads only for courses present in their signed claims.
- Message sender role and name must match the signed Firebase identity.
- Storage rules enforce the same course/thread access and a 5 MB image limit.
- Admin Firebase access remains intentionally disabled until admin LMS
  integration is implemented.

Rich Media Chat
===============

- Students and teachers can choose photos from the gallery.
- Camera photos are captured and sent immediately.
- Existing videos can be selected and sent.
- Camera videos can be recorded for up to two minutes and sent immediately.
- Voice messages use tap-to-record and tap-to-stop-and-send.
- Voice recordings can be cancelled and auto-stop after five minutes.
- Voice messages include inline playback and duration progress.
- Videos include inline playback controls.
- Upload limits are enforced in Flutter and Firebase Storage:
  - Images: 5 MB
  - Voice: 10 MB
  - Videos: 50 MB
- Android and iOS camera, microphone, and photo-library permissions are
  configured.
- Firestore and Storage rules support image, voice, and video messages.

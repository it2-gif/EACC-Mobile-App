# EACC Chat Production Architecture

Status: Proposed  
Last updated: 2026-06-18

## 1. Purpose

This document defines the production data model and service boundaries for EACC
Chat. It is the contract for the NestJS backend, PostgreSQL database, Firebase
Authentication, Firestore, Firebase Storage, Firebase Cloud Messaging, LMS
integration, and Flutter application.

The design supports three roles:

- `student`
- `teacher`
- `admin`

## 2. Architecture Decisions

### Sources of truth

| Data | Source of truth |
| --- | --- |
| Users and roles | PostgreSQL |
| Courses | PostgreSQL |
| Course memberships | PostgreSQL |
| LMS synchronization state | PostgreSQL |
| Live chat threads and messages | Firestore |
| Chat attachments | Firebase Storage |
| Device notification tokens | PostgreSQL |
| Admin and security audit events | PostgreSQL |
| Pending Firebase synchronization work | PostgreSQL outbox |

Firestore authorization documents are generated from PostgreSQL membership
data. They are access-control projections, not an independent membership source.

### Important rules

- Use internal UUIDs as application IDs.
- Store LMS IDs separately and enforce uniqueness by LMS source.
- Set the Firebase Authentication `uid` to the internal user UUID.
- Never use a person's name as an identity or document key.
- Do not store chat messages in both Firestore and PostgreSQL initially.
- Do not let Flutter choose or change the authenticated user's role.
- The backend validates LMS responses and controls Firebase custom claims.

## 3. High-Level Flow

```text
Flutter
  |
  | POST /v1/auth/lms-login
  v
NestJS Backend
  |
  +--> EACC LMS: validate credentials and load user/course data
  |
  +--> PostgreSQL: upsert user, courses, and memberships
  |
  +--> Firestore: synchronize course/member authorization documents
  |
  +--> Firebase Admin SDK: create custom authentication token
  |
  v
Flutter signs in to Firebase with the custom token
  |
  +--> Firestore real-time chat
  +--> Firebase Storage attachments
  +--> Firebase Cloud Messaging notifications
```

## 4. PostgreSQL Model

Use PostgreSQL UUID primary keys generated by the database or Prisma.
Use `timestamptz` for all timestamps and store all times in UTC.

### Enums

```text
user_role: student | teacher | admin
user_status: active | suspended | disabled
course_status: active | archived
membership_status: active | inactive
device_platform: android | ios | web
audit_result: success | failure
```

### `users`

Stores the permanent application identity.

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key; also used as Firebase `uid` |
| `lms_source` | VARCHAR(50) | Default `eacc_lms` |
| `lms_user_id` | VARCHAR(100) | Required |
| `role` | `user_role` | Required |
| `name` | VARCHAR(200) | Required |
| `email` | VARCHAR(320) | Nullable until confirmed by LMS |
| `status` | `user_status` | Default `active` |
| `last_login_at` | TIMESTAMPTZ | Nullable |
| `created_at` | TIMESTAMPTZ | Required |
| `updated_at` | TIMESTAMPTZ | Required |

Constraints and indexes:

- Unique: `lms_source`, `lms_user_id`, `role`
- Index: `email`
- Index: `role`, `status`

The role is part of the LMS identity key because some legacy LMS systems can
reuse numeric IDs across student and teacher tables.

### `courses`

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `lms_source` | VARCHAR(50) | Default `eacc_lms` |
| `lms_course_id` | VARCHAR(100) | Required |
| `name` | VARCHAR(250) | Required |
| `category` | VARCHAR(200) | Nullable |
| `status` | `course_status` | Default `active` |
| `starts_at` | TIMESTAMPTZ | Nullable |
| `ends_at` | TIMESTAMPTZ | Nullable |
| `created_at` | TIMESTAMPTZ | Required |
| `updated_at` | TIMESTAMPTZ | Required |

Constraints and indexes:

- Unique: `lms_source`, `lms_course_id`
- Index: `status`
- Index: `name`

### `course_memberships`

Represents student enrollment, teacher assignment, and optional scoped admin
access.

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `course_id` | UUID | Foreign key to `courses.id` |
| `user_id` | UUID | Foreign key to `users.id` |
| `role` | `user_role` | Must match the membership purpose |
| `status` | `membership_status` | Default `active` |
| `synced_at` | TIMESTAMPTZ | Last confirmed by LMS |
| `created_at` | TIMESTAMPTZ | Required |
| `updated_at` | TIMESTAMPTZ | Required |

Constraints and indexes:

- Unique: `course_id`, `user_id`, `role`
- Index: `user_id`, `status`
- Index: `course_id`, `role`, `status`

Global admins do not require one membership row per course. Their global access
comes from the trusted Firebase `role=admin` custom claim. Add scoped admin
memberships later only if the business requires restricted administrators.

### `device_tokens`

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `user_id` | UUID | Foreign key to `users.id` |
| `token` | TEXT | Required |
| `platform` | `device_platform` | Required |
| `device_name` | VARCHAR(200) | Nullable |
| `active` | BOOLEAN | Default `true` |
| `last_seen_at` | TIMESTAMPTZ | Required |
| `created_at` | TIMESTAMPTZ | Required |
| `updated_at` | TIMESTAMPTZ | Required |

Constraints and indexes:

- Unique: `token`
- Index: `user_id`, `active`

### `audit_logs`

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `actor_user_id` | UUID | Nullable foreign key to `users.id` |
| `action` | VARCHAR(100) | Required |
| `resource_type` | VARCHAR(100) | Required |
| `resource_id` | VARCHAR(200) | Nullable |
| `result` | `audit_result` | Required |
| `ip_address` | INET | Nullable |
| `user_agent` | TEXT | Nullable |
| `metadata` | JSONB | Default empty object |
| `created_at` | TIMESTAMPTZ | Required and immutable |

Indexes:

- Index: `actor_user_id`, `created_at`
- Index: `action`, `created_at`
- Index: `resource_type`, `resource_id`

Never store passwords, Firebase tokens, notification tokens, or full sensitive
LMS responses inside audit metadata.

### `lms_sync_runs`

Tracks synchronization health and makes LMS failures diagnosable.

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `sync_type` | VARCHAR(50) | Login, course, or scheduled sync |
| `status` | VARCHAR(30) | Started, completed, or failed |
| `records_read` | INTEGER | Default `0` |
| `records_changed` | INTEGER | Default `0` |
| `error_code` | VARCHAR(100) | Nullable |
| `started_at` | TIMESTAMPTZ | Required |
| `finished_at` | TIMESTAMPTZ | Nullable |

### `integration_outbox`

Provides reliable PostgreSQL-to-Firebase synchronization. The backend writes
business changes and their outbox events in the same database transaction. A
worker retries Firebase updates until they succeed.

| Column | Type | Rules |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `event_type` | VARCHAR(100) | Required |
| `aggregate_type` | VARCHAR(100) | Required |
| `aggregate_id` | UUID | Required |
| `payload` | JSONB | Required |
| `attempts` | INTEGER | Default `0` |
| `available_at` | TIMESTAMPTZ | Required |
| `processed_at` | TIMESTAMPTZ | Nullable |
| `last_error_code` | VARCHAR(100) | Nullable |
| `created_at` | TIMESTAMPTZ | Required |

Indexes:

- Index: `processed_at`, `available_at`
- Index: `aggregate_type`, `aggregate_id`

Suggested event types:

- `user.projection.upsert`
- `course.projection.upsert`
- `membership.projection.upsert`
- `membership.projection.deactivate`

Outbox payloads must not contain passwords, Firebase tokens, or unnecessary LMS
response data.

## 5. Relationship Summary

```text
users 1 --- * course_memberships * --- 1 courses
users 1 --- * device_tokens
users 1 --- * audit_logs

PostgreSQL user.id == Firebase Authentication uid
PostgreSQL course.id == Firestore courses/{courseId} document ID
PostgreSQL student user.id == Firestore thread document ID
```

A thread ID is the student UUID. This gives each student exactly one thread per
course and prevents duplicate thread creation.

## 6. Firestore Model

### `users/{userId}`

Authorization/profile projection written by the backend.

```json
{
  "name": "Esam Test",
  "role": "student",
  "status": "active",
  "updated_at": "server timestamp"
}
```

The document ID must equal the Firebase Authentication `uid`.

### `courses/{courseId}`

```json
{
  "name": "Elementary Level - 3",
  "category": "English Adult",
  "status": "active",
  "lms_course_id": "2191",
  "updated_at": "server timestamp"
}
```

The document ID must equal `courses.id` from PostgreSQL, not the course name.

### `courses/{courseId}/members/{userId}`

Authorization projection written only by the backend.

```json
{
  "role": "student",
  "active": true,
  "updated_at": "server timestamp"
}
```

Teachers receive `role: "teacher"`. Students receive `role: "student"`.

### `courses/{courseId}/threads/{studentId}`

The thread document ID is the student's internal UUID.

```json
{
  "student_id": "user UUID",
  "student_name": "Esam Test",
  "last_message": "Hello teacher",
  "last_message_type": "text",
  "last_sender_id": "user UUID",
  "last_sender_name": "Esam Test",
  "last_sender_role": "student",
  "last_message_at": "server timestamp",
  "created_at": "server timestamp",
  "updated_at": "server timestamp"
}
```

Names are display snapshots. Authorization always uses IDs and membership
documents.

### `courses/{courseId}/threads/{studentId}/messages/{messageId}`

```json
{
  "sender_id": "user UUID",
  "sender_name": "Esam Test",
  "sender_role": "student",
  "type": "text",
  "text": "Hello teacher",
  "attachment": null,
  "created_at": "server timestamp"
}
```

Image message:

```json
{
  "sender_id": "user UUID",
  "sender_name": "Esam Test",
  "sender_role": "student",
  "type": "image",
  "text": "",
  "attachment": {
    "storage_path": "chat_uploads/courses/.../image.jpg",
    "file_name": "image.jpg",
    "content_type": "image/jpeg",
    "size_bytes": 245678
  },
  "created_at": "server timestamp"
}
```

Do not treat a public download URL as the permanent attachment identity. Store
the Storage path and resolve access through Firebase Storage.

### Message validation

- `sender_id` must equal `request.auth.uid`.
- `sender_role` must equal the trusted Firebase role claim.
- `type` must be an allowed value.
- Text messages require non-empty trimmed text.
- Attachment metadata must match the allowed type.
- `created_at` must be a server timestamp.
- Existing messages should be immutable initially.
- Message deletion and editing require a separately designed policy.

### Firestore indexes

Required query patterns:

- Threads ordered by `updated_at` descending.
- Messages ordered by `created_at` descending with pagination.

The current subcollection queries usually use automatic single-field indexes.
Add composite indexes only when Firebase reports a specific required index.

## 7. Firebase Storage Model

Production path:

```text
chat_uploads/courses/{courseId}/threads/{studentId}/messages/{messageId}/{fileName}
```

Using the message ID in the path gives every attachment a stable owner and
prevents collisions.

Initial limits:

| Type | Allowed formats | Maximum size |
| --- | --- | --- |
| Image | JPG, JPEG, PNG, WEBP, GIF | 5 MB |
| Video | Define before implementation | Not approved yet |
| File | Define before implementation | Not approved yet |
| Audio | Define before implementation | Not approved yet |

Storage authorization must verify:

- The caller is authenticated and active.
- A student owns the thread and is an active course member.
- A teacher is an active teacher member of the course.
- An admin has the trusted global admin claim.
- The uploaded content type and size are allowed.

## 8. Firebase Authentication

### Custom token identity

Firebase custom token:

```json
{
  "uid": "internal PostgreSQL user UUID",
  "claims": {
    "role": "student",
    "status": "active"
  }
}
```

Do not place course membership arrays in custom claims. They can exceed token
limits and become stale. Firestore membership documents determine course
access.

### Token refresh

When the backend changes a user's role or status:

1. Update PostgreSQL.
2. Update the Firestore user projection and memberships.
3. Revoke Firebase refresh tokens when access must end immediately.
4. Require Flutter to authenticate again.

## 9. Login API Contract

### `POST /v1/auth/lms-login`

Request:

```json
{
  "username": "student@example.com",
  "password": "user-entered password",
  "device": {
    "platform": "android",
    "name": "Samsung A54"
  }
}
```

Rules:

- Use HTTPS only.
- Never log or store the password.
- Apply rate limiting by IP and normalized username.
- The backend determines the role from the LMS response.
- Remove the role selector from Flutter production login.

Successful response:

```json
{
  "firebase_custom_token": "token",
  "user": {
    "id": "user UUID",
    "lms_user_id": "55",
    "name": "Esam Test",
    "email": "student@example.com",
    "role": "student",
    "status": "active"
  },
  "courses": [
    {
      "id": "course UUID",
      "lms_course_id": "2191",
      "name": "Elementary Level - 3",
      "category": "English Adult",
      "membership_role": "student"
    }
  ]
}
```

Error response:

```json
{
  "status_code": 401,
  "code": "INVALID_CREDENTIALS",
  "message": "The username or password is incorrect."
}
```

Recommended error codes:

- `INVALID_CREDENTIALS`
- `ACCOUNT_DISABLED`
- `LMS_UNAVAILABLE`
- `LMS_RESPONSE_INVALID`
- `RATE_LIMITED`
- `INTERNAL_ERROR`

Do not return raw LMS errors or stack traces to Flutter.

### `GET /v1/me`

Returns the current user and current course memberships after validating a
Firebase ID token.

### `POST /v1/devices`

Registers or refreshes an FCM device token.

```json
{
  "token": "fcm token",
  "platform": "android",
  "device_name": "Samsung A54"
}
```

### `DELETE /v1/devices/{token}`

Disables a notification token during logout.

## 10. LMS Mapping

The backend must adapt the real LMS response into one normalized structure.
Flutter must never parse multiple LMS-specific student/teacher response shapes.

### Student mapping

| LMS field | Production field |
| --- | --- |
| `st_id` | `users.lms_user_id` |
| `st_name` | `users.name` |
| `st_email` | `users.email` |
| Student response type | `users.role = student` |
| `course_id` | `courses.lms_course_id` |
| `course_name` | `courses.name` |
| `category` | `courses.category` |

### Teacher mapping

| LMS field | Production field |
| --- | --- |
| `te_id` | `users.lms_user_id` |
| `te_name` | `users.name` |
| `te_email` | `users.email` |
| Teacher response type | `users.role = teacher` |
| `course_id` | `courses.lms_course_id` |
| Course students | Student users and memberships |

### Synchronization rules

- Upsert users and courses in one PostgreSQL transaction.
- Upsert active memberships returned by the LMS.
- Mark missing memberships inactive only when the LMS response is complete and
  authoritative.
- Do not delete users, courses, memberships, or chat history during normal sync.
- Write Firestore projection events to `integration_outbox` in the same
  PostgreSQL transaction.
- Let an idempotent worker apply and retry Firestore projection changes.
- Record synchronization failures in `lms_sync_runs`.

## 11. Permission Matrix

| Action | Student | Teacher | Admin |
| --- | --- | --- | --- |
| Read own profile | Yes | Yes | Yes |
| Read another user's profile | No | Limited course display data | Yes |
| List assigned courses | Yes | Yes | All courses |
| Read own course thread | Yes | Yes, when assigned | Yes |
| Read another student's thread | No | Yes, when assigned | Yes |
| Create own thread | Yes | No | No |
| Send message in accessible thread | Yes | Yes | Yes |
| Upload attachment in accessible thread | Yes | Yes | Yes |
| Change membership | No | No | Through backend only |
| Change role | No | No | Through backend only |
| Suspend user | No | No | Through backend only |
| Read audit logs | No | No | Authorized admins only |

Admin chat presentation is a UI rule: all messages remain left-aligned for
monitoring. It is separate from backend authorization.

## 12. Firestore Rule Strategy

Helper concepts:

```text
isSignedIn:
  request.auth != null

isAdmin:
  request.auth.token.role == "admin"

isActiveUser:
  users/{request.auth.uid}.status == "active"

isActiveMember(courseId):
  courses/{courseId}/members/{request.auth.uid}.active == true

isTeacher(courseId):
  member role == "teacher"

ownsThread(studentId):
  request.auth.uid == studentId
```

Rules should enforce:

- Only the backend can write `users`, `courses`, and `members` projections.
- Every chat or attachment operation requires an active user projection.
- Students can read only their own thread.
- Teachers can read threads only in assigned courses.
- Admins can read all chat documents.
- Message creates must use the authenticated sender ID and role.
- Clients cannot change sender identity after creation.
- Clients cannot update membership or role documents.

Firebase Admin SDK operations bypass Firestore rules. Backend endpoints must
therefore perform their own authorization checks.

## 13. Notification Flow

1. Flutter registers an FCM token through `POST /v1/devices`.
2. A new Firestore message triggers a trusted notification worker.
3. The worker identifies recipients using PostgreSQL memberships.
4. The sender is excluded.
5. Notification data includes only IDs needed for navigation:

```json
{
  "type": "chat_message",
  "course_id": "course UUID",
  "thread_id": "student UUID",
  "message_id": "message ID"
}
```

Avoid putting message text or sensitive student data in notification payloads
unless EACC explicitly approves that privacy behavior.

## 14. Migration from the Current Prototype

Current prototype IDs such as `2191` and `student_1` are development IDs.

Migration order:

1. Create PostgreSQL users, courses, and memberships.
2. Generate internal UUIDs.
3. Create Firebase Authentication users/custom-token identities.
4. Create Firestore user, course, and member projections.
5. Create new UUID-based thread paths.
6. Optionally copy development messages into the new paths.
7. Verify student, teacher, and admin access with Firebase Emulator tests.
8. Deploy secure rules only after the Flutter app uses authenticated UUIDs.
9. Remove fake role selection and `fake_data.dart` usage.

Development messages may be discarded instead of migrated if they contain only
test data.

## 15. Implementation Order

1. Approve this architecture and confirm the real LMS response formats.
2. Create the NestJS backend in a separate `backend/` directory or repository.
3. Configure Prisma and PostgreSQL.
4. Implement the enums and tables in this document.
5. Add Firebase Admin SDK configuration through environment variables.
6. Implement LMS client adapters and `/v1/auth/lms-login`.
7. Add PostgreSQL-to-Firestore projection synchronization.
8. Replace Flutter fake login with backend login and Firebase custom sign-in.
9. Replace fake courses/users with authenticated data.
10. Write Firestore and Storage emulator rule tests.
11. Deploy secure rules to staging.
12. Add FCM device registration and notifications.

## 17. Open Questions Before LMS Implementation

These facts must be confirmed with the LMS owner:

1. What is the exact login endpoint and method?
2. Does one account ever have more than one role?
3. Are student and teacher IDs unique across role tables?
4. Does the login response contain the complete course list?
5. How are removed enrollments represented?
6. Are course IDs stable across terms?
7. Is there an admin login endpoint?
8. Are rate limits or IP restrictions applied?
9. What timeout and retry behavior is acceptable?
10. May the backend cache non-sensitive LMS profile/course data?

Do not begin production LMS parsing until sample success and failure responses
for every role are available.
